# Spine Tier 1 — Data Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Stand up the ADIP multi-tenant data foundation — repo scaffold, Supabase schema for the thin spine, JWT tenant claims, and RLS policies proven to block cross-tenant reads.

**Architecture:** Supabase-local-first (CLI + Docker). All spine tables carry `tenant_id`. A Custom Access Token Hook injects `tenant_id` + `role` into the JWT. RLS policies on every table key off `auth.tenant_id()` (a helper reading the JWT claim) — RLS is the security boundary, app filtering is supplementary. Isolation is verified by a pgTAP test that sets JWT claims for tenant A and asserts zero visibility of tenant B's rows.

**Tech Stack:** Supabase CLI · PostgreSQL · pgTAP · SQL migrations.

**Reference:** `docs/specs/2026-06-07-spine-design.md` §4–§9. Source-driven note: verify Supabase CLI command surface and the Custom Access Token Hook signature against current Supabase docs during execution (`supabase` CLI help + Auth Hooks docs) — the SQL below targets the stable PostgREST/GoTrue claim conventions.

```sql
-- 0005_rls_helpers.sql — read tenant_id/role from the JWT claims

-- Returns the tenant_id claim from the current request's JWT, or null.
create or replace function auth.tenant_id()
returns uuid
language sql stable
as $$
  select nullif(
    current_setting('request.jwt.claims', true)::jsonb ->> 'tenant_id',
    ''
  )::uuid;
$$;

-- Returns the role claim (app role, not the Postgres role) from the JWT.
create or replace function auth.user_role()
returns text
language sql stable
as $$
  select current_setting('request.jwt.claims', true)::jsonb ->> 'user_role';
$$;
```

## Task 8: Custom Access Token Hook (JWT claims)

**Files:**

- Modify: `supabase/config.toml` (enable the hook)

- [ ] **Step 1: Write the hook migration**

```sql
-- 0006_jwt_claims_hook.sql — inject tenant_id + user_role into the access token
-- Signature per Supabase Custom Access Token Hook: receives + returns jsonb event.

create or replace function public.custom_access_token_hook(event jsonb)
returns jsonb
language plpgsql stable
as $$
declare
  claims    jsonb;
  v_tenant  uuid;
  v_role    text;
begin
  select u.tenant_id, u.role::text
    into v_tenant, v_role
    from public.users u
   where u.id = (event ->> 'user_id')::uuid;

  claims := event -> 'claims';

  if v_tenant is not null then
    claims := jsonb_set(claims, '{tenant_id}', to_jsonb(v_tenant::text));
    claims := jsonb_set(claims, '{user_role}', to_jsonb(v_role));
  end if;

  event := jsonb_set(event, '{claims}', claims);
  return event;
end;
$$;

-- The Auth admin role must be able to execute the hook.
grant execute on function public.custom_access_token_hook(jsonb) to supabase_auth_admin;
grant select on public.users to supabase_auth_admin;
revoke execute on function public.custom_access_token_hook(jsonb) from authenticated, anon, public;
```

- [ ] **Step 2: Enable the hook in `config.toml`**

Add (or update) this block in `supabase/config.toml`:


## Task 9: RLS policies on every domain table

**Files:**
- Create: `supabase/migrations/0007_rls_policies.sql`

- [ ] **Step 1: Write the migration**

```sql
-- 0007_rls_policies.sql — RLS is the tenant-isolation boundary.

-- Enable RLS on all domain tables.
alter table public.tenants        enable row level security;
alter table public.locations      enable row level security;
alter table public.users          enable row level security;
alter table public.user_locations enable row level security;
alter table public.customers      enable row level security;
alter table public.agent_registry enable row level security;
alter table public.notifications  enable row level security;
alter table public.audit_logs     enable row level security;

-- tenants: a user sees only their own tenant row.
create policy tenant_isolation_select on public.tenants
  for select to authenticated
  using (id = auth.tenant_id());

-- Generic tenant_id-scoped tables: full access within own tenant.
create policy loc_tenant on public.locations
  for all to authenticated
  using (tenant_id = auth.tenant_id()) with check (tenant_id = auth.tenant_id());

create policy users_tenant on public.users
  for all to authenticated
  using (tenant_id = auth.tenant_id()) with check (tenant_id = auth.tenant_id());

create policy customers_tenant on public.customers
  for all to authenticated
  using (tenant_id = auth.tenant_id()) with check (tenant_id = auth.tenant_id());

create policy notifications_tenant on public.notifications
  for all to authenticated
  using (tenant_id = auth.tenant_id()) with check (tenant_id = auth.tenant_id());

create policy audit_tenant on public.audit_logs
  for all to authenticated
  using (tenant_id = auth.tenant_id()) with check (tenant_id = auth.tenant_id());

-- user_locations: scoped via the joined user's tenant.
create policy user_locations_tenant on public.user_locations
  for all to authenticated
  using (exists (
    select 1 from public.users u
     where u.id = user_locations.user_id and u.tenant_id = auth.tenant_id()
  ))
  with check (exists (
    select 1 from public.users u
     where u.id = user_locations.user_id and u.tenant_id = auth.tenant_id()
  ));

-- agent_registry: global agents (tenant_id null) are visible to all; tenant agents scoped.
create policy agents_visibility on public.agent_registry
  for select to authenticated
  using (tenant_id is null or tenant_id = auth.tenant_id());
create policy agents_write on public.agent_registry
  for all to authenticated
  using (tenant_id = auth.tenant_id()) with check (tenant_id = auth.tenant_id());
```

This pgTAP test impersonates an `authenticated` user from tenant ABC (by setting the JWT claims GUC) and asserts it can see ABC's customers but **zero** of XYZ's.

```sql
-- isolation_test.sql — RLS must block cross-tenant reads.
begin;
select plan(3);

-- Impersonate an authenticated request from tenant ABC.
set local role authenticated;
set local request.jwt.claims = '{"role":"authenticated","tenant_id":"11111111-1111-1111-1111-111111111111","user_role":"dealer_owner"}';

-- 1. ABC sees its own 2 customers.
select is(
  (select count(*)::int from public.customers),
  2,
  'ABC sees exactly its own 2 customers'
);

-- 2. ABC sees zero of XYZ's customers explicitly.
select is(
  (select count(*)::int from public.customers
     where tenant_id = '22222222-2222-2222-2222-222222222222'),
  0,
  'ABC cannot see XYZ customers'
);

-- 3. ABC sees only its own tenant row.
select is(
  (select count(*)::int from public.tenants),
  1,
  'ABC sees only its own tenant row'
);

select * from finish();
rollback;
```

## Post-build fix — the auth seam (Task 12, verified 2026-06-07)
The original plan verified RLS with *manually-set* JWT claims but never exercised the real login→hook→claims path. Doing so revealed a blocking bug and two gaps:
- **Bug:** `custom_access_token_hook` reads `public.users` as `supabase_auth_admin`, but RLS was enabled on `public.users` with no policy for that role → hook read 0 rows → tenant-less JWTs → RLS denied real users everything. **Fix:** `0008_auth_admin_users_policy.sql` adds `create policy auth_admin_read_users on public.users for select to supabase_auth_admin using (true)`.
- **Gap:** seed had no users. **Fix:** `scripts/seed_demo_users.py` (idempotent, HTTP-only) creates 3 demo accounts + their `public.users` rows after `supabase db reset`.
- **Verified:** real password logins for ABC owner (sees 2 customers) and XYZ sales (sees 1) return JWTs carrying `tenant_id`+`user_role`; PostgREST reads respect RLS. AC#1 and AC#2 now satisfied.

## As-built deviations (verified 2026-06-07)
- **Helpers live in `public`, not `auth`.** Recent Supabase locks the `auth` schema (`permission denied for schema auth`). Functions are `public.tenant_id()` / `public.user_role()`; all RLS policies reference `public.tenant_id()`. **Tier 2/3 must use `public.tenant_id()`.**
- **`[analytics] enabled = false` in `config.toml`.** The `vector` log-collector mounts the Docker socket, which colima does not support (`operation not supported`). Analytics/logflare is unused by the data layer; disabling it lets the stack boot.
- **Runtime:** colima (not Docker Desktop). `colima start` before `supabase start`.
