> Generated by analyzing the codebase as it stands today. This document
> describes **what actually exists and runs**, as distinct from
> `ADIP_MASTER_PLAN.md` (the intended end-state) and `ARCHITECTURE_RULES.md`
> (the rules the codebase is supposed to follow). Where reality diverges from
> either, it is called out explicitly rather than smoothed over.
>
> **Last updated:** after Phase 7 (Event-Driven Architecture) вҖ” complete (see В§21).
> Phases 1вҖ“7 are all complete. Agents are now decoupled: they communicate through
> an in-process event bus (`domain_events`) rather than direct calls вҖ” the
> intakeвҶ’workflow, callвҶ’re-score, re-scoreвҶ’workflow, and WhatsApp-replyвҶ’re-score

**Runtime topology (local dev, `npm run dev`):**

| Process | Port | Role |
|---|---|---|
| DuckDB Shim (Express) | 54321 | PostgREST + Auth emulation + SSE hub |
| TanStack Start (Vite) | 3000 | Frontend + BFF server fns |
| FastAPI (uvicorn) | 8000 | Agent orchestration, business logic |
| ngrok (optional) | вҖ” | HTTPS tunnel вҶ’ port 8000 for Meta webhooks |

The browser's `EventSource` connects to **port 54321** (the shim), not to FastAPI directly.
FastAPI notifies the shim of events via internal HTTP POSTs when something happens that
the browser needs to know about immediately (e.g. delivery status changes).

```
  в”ңв”Җв”Җ Error body captured and logged; raises ValueError on non-2xx
  в”ңв”Җв”Җ send_url_media: uses {"link": url} not {"id": media_id} вҖ” no prior upload needed
  в””в”Җв”Җ verify_signature(body, sig): HMAC-SHA256 via WHATSAPP_APP_SECRET
        в””в”Җв–ә if WHATSAPP_APP_SECRET unset вҶ’ accepts all (warns in log, local dev safe)

MockWhatsAppProvider
  в”ңв”Җв”Җ Returns {"wamid": "mock-wamid-{uuid12}", "status": "sent"}
  в”ңв”Җв”Җ Logs to console, no real HTTP call
  в””в”Җв”Җ Identical DB writes as real provider (same lead_messages + message_delivery_logs rows)

Fallback chain in send_message_node:
  1. get_provider() (Meta or Mock per env)
  2. On any exception вҶ’ MockWhatsAppProvider (always succeeds)
  3. On mock failure вҶ’ wamid=None, errors recorded, no DB write
  provider_used: "meta" | "mock" | "mock_fallback"
```


**Why the shim is the SSE hub:** The browser's `EventSource` connects to
`VITE_SUPABASE_URL` (the shim at port 54321) because the web app is designed
to work against both real Supabase and the local shim without code changes.
FastAPI (port 8000) is not the SSE hub. This creates the "reversed direction"
problem for agent-triggered events: FastAPI does the work but the browser only
listens to the shim. Solution: FastAPI POSTs to the shim after every significant
event (`/events/whatsapp-status`, `/events/whatsapp-inbound`, `/events/rescore-complete`),
mirroring the Phase 2 `stage_change` pattern. All six event types share the same
`EventSource` connection the browser opens to `{VITE_SUPABASE_URL}/intake/stream`.

Rule table: `hot вҶ’ [call, whatsapp, manager_escalation]`, `warm вҶ’ [whatsapp]`,
`cold вҶ’ [nurture]`, `dead вҶ’ [close]`. Override: if `test_drive_required=true`
and no prior `test_drive` event exists, `test_drive` is appended regardless of score.

**WhatsApp Agent** вң… Phase 4 вҖ” linear, always completes:
```
load_context вҶ’ send_message вҶ’ log_delivery вҶ’ END
```
`send_message` never raises: tries primary provider, falls back to `MockWhatsAppProvider`
on any exception, records the failure in `errors` but still produces a `wamid`.
`log_delivery` is best-effort: a DB failure records itself in `errors` but does not
re-raise or affect the response.

**Call Intelligence Agent** вң… Phase 5 вҖ” linear, always completes:
```
transcribe вҶ’ extract вҶ’ persist вҶ’ handoff вҶ’ END
```
Every node is failure-tolerant (records to `errors`, never raises). `transcribe`
runs faster-whisper via `run_in_executor` (CPU-bound, off the event loop) with a
mock-transcript fallback if the model/lib is unavailable. `extract` makes ONE Groq
call, parses+clamps the JSON, and falls back to a deterministic keyword analysis.
`persist` UPSERTs `call_analysis` by `call_id` (idempotent retries). `handoff` POSTs
to `/rescore/{lead_id}` and degrades silently if Phase 6 is unreachable.

## 21. Phase 7 вҖ” Event-Driven Architecture

Removes direct agent-to-agent coupling (ARCHITECTURE_RULES #4). Agents now
**publish/consume domain events** through an in-process async bus. No external
broker вҖ” that's deliberately deferred (a broker belongs to Phase 8 at the
earliest); the bus's transport can be swapped later without touching producers
or handlers.

**New package `agents/events/`:**
- `types.py` вҖ” `EventType` (the 12 events) + `DomainEvent` envelope (`id`,
  `tenant_id`, `lead_id`, `payload`, `source`, `created_at`).
- `store.py` вҖ” `EventStore`: persists/updates/lists `domain_events` over
  PostgREST (best-effort; a store hiccup never blocks delivery).
- `bus.py` вҖ” `EventBus`: `subscribe()` / `publish()`; persists each event, then
  dispatches to every subscriber as a background task with **retry + backoff**,
  marking the row `done`/`failed`. `replay()` re-dispatches non-`done` rows.

**Producer вҶ’ event вҶ’ subscriber map (handlers wrap existing agents, unchanged):**

| Event | Published by | Subscriber вҶ’ agent |
|---|---|---|
| `LEAD_ASSIGNED` | intake (`/intake/leads`) | `_on_lead_assigned` вҶ’ Workflow Agent |
| `CALL_COMPLETED` | Call Intelligence `handoff_node` | `_on_call_completed` вҶ’ Re-Scoring |
| `MESSAGE_READ` | WhatsApp inbound webhook | `_on_message_read` вҶ’ Re-Scoring |

| `TEST_DRIVE_BOOKED` | (stage-change, accepted) | `_on_test_drive_booked` вҶ’ Re-Scoring |

| `LEAD_CREATED/VALIDATED/SCORED`, `MESSAGE_SENT`, `SENTIMENT_UPDATED`, `LEAD_CLOSED` | (reserved) | persisted for observability; no behavioural subscriber yet |

The full chain now flows through events:
`intake вҶ’ LEAD_ASSIGNED вҶ’ Workflow`; `call вҶ’ CALL_COMPLETED вҶ’ Re-Score вҶ’
LEAD_RESCORED вҶ’ Workflow`; `WhatsApp reply вҶ’ MESSAGE_READ вҶ’ Re-Score вҶ’ вҖҰ`.

**Design properties:**
- **Observable** вҖ” every event is a row in `domain_events` (`GET /events`).
- **Recoverable** вҖ” `replay()` on startup + `POST /events/replay` re-dispatch
  non-`done` events (persist-before-dispatch).
- **Resilient** вҖ” handler failures are isolated, retried (`EVENT_MAX_RETRIES`,
  `EVENT_RETRY_BACKOFF_MS`), logged, and never block the producer (rule #13/#14).
- **Reversible kill-switch** вҖ” `EVENT_BUS_ENABLED=0` makes `publish()` dispatch
  handlers **inline** (no persistence/retry) вҖ” i.e. legacy-like direct behaviour
  through the same registry, so producers need no fallback branch.

**Backward compatibility:**
- Agent service functions (`run_workflow_agent`, `rescore_lead`, `process_call`)
  are **unchanged** вҖ” handlers wrap them. All 38 prior tests still pass; 8 new
  bus tests added (**46 total**).
- HTTP endpoints (`/workflow`, `/rescore`, `/followup`, `/whatsapp/*`) and the
  shim's `triggerX` calls still work вҖ” they become event **producers**. The JS
  shim is a dev-only gateway (not an agent), so its HTTP triggers remain an
  accepted external producer; full shimвҶ’event migration is a follow-up.

**Env vars:** `EVENT_BUS_ENABLED` (default `1`), `EVENT_MAX_RETRIES` (default `3`),
`EVENT_RETRY_BACKOFF_MS` (default `500`). All optional with safe defaults.

**Known follow-ups:** `LEAD_CREATED/VALIDATED/SCORED`, `SENTIMENT_UPDATED`,
`MESSAGE_SENT`, `LEAD_CLOSED` are defined and can be emitted but have no
behavioural subscriber yet; the shim still triggers via HTTP rather than
publishing; in-process bus is not durable across a crash mid-dispatch (mitigated
by persist-before-dispatch + replay вҖ” a real broker is a Phase 8 consideration).
